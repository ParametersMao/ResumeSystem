1. 概述
   本文档旨在描述一个用于生成个人简历或信息展示的 Web 应用的原型设计。该应用的核心功能是允许用户通过模块化的方式编辑个人信息，并实时预览最终效果，最终能够导出为 PDF 等格式。整个系统分为前端和后端两个主要部分，通过 JSON 格式的数据进行通信和交互。

2. 系统架构
   2.1 整体架构
   前端: 负责用户界面（UI）的展示、用户输入的处理、实时预览的渲染以及与后端的数据交互。

后端: 负责数据的存储、模板的管理、以及根据前端请求生成最终的简历文件（如 PDF）。

数据通信: 前后端之间通过 JSON 格式进行数据交换。前端将用户编辑的数据以 JSON 格式发送给后端，后端则返回相应的 JSON 数据或处理结果。

2.2 数据结构
应用的核心数据是模块化的 JSON 结构。为了支持模块的排序，我们将所有模块数据放在一个有序的数组中。为了实现精美的预设模板，我们为每个模块添加了独立的 style 字段，该字段可以定义模块的布局和样式。

3. 功能模块
   3.1 前端功能模块
   3.1.1 模块化编辑区域
   这是用户进行内容编辑的主要区域，它包含了与所有模板都一一对应的固定模块。

固定模块列表: 界面中将默认显示以下所有模块，其顺序和排版信息由模板 JSON 定义：

基本信息 (Basic Info)

求职意向 (Job Intent)

教育背景 (Education)

工作经验 (Work Experience)

项目经历 (Project Experience)

实习经历 (Internship Experience)

校园经历 (Campus Experience)

技能特长 (Skills & Strengths)

荣誉证书 (Honors & Certifications)

自我评价 (Self-Evaluation)

兴趣爱好 (Hobbies)

自定义模块 (Custom Module)

模块名称可编辑: 在模块化编辑区域，每个模块的标题都将是一个可编辑的输入框。用户可以点击标题进行编辑，编辑后的内容将同步更新到 JSON 数据的 title 字段中。

模块排序: 在模块编辑区域，每个模块都应该提供“上移”和“下移”按钮，允许用户自由调整模块的顺序。模块在预览区域的显示顺序将严格遵循编辑区域的排序。

模块内部条目排序: 对于那些可重复添加的模块（如教育背景、工作经验等），其内部的每个条目（items 数组中的每个对象）也应提供“上移”和“下移”按钮，允许用户调整其在模块内的显示顺序。

富文本编辑框: 允许用户输入复杂格式的内容，如工作经历、项目经历等。前端将采用 WangEditor 作为富文本编辑器。

模块管理: 底部提供“新增教育背景”等按钮，用户可以添加新的可重复模块条目。

3.1.2 即时预览区域
功能: 实时显示用户在编辑区域输入的内容，所见即所得。

数据流: 当编辑区域的数据发生变化时，前端立即更新预览区域的内容，使其与最终渲染效果一致。

样式渲染: 前端将根据 JSON 数据中 globalStyles 和每个模块自己的 style 字段来渲染预览区域。模块的 style 字段可以覆盖 globalStyles 中的一些默认样式，实现定制化的布局。例如，通过解析 style 字段中的 gridColumn 属性，可以将模块放置在网格布局的不同列中。

AI 润色: 可能包含 AI 润色功能，润色后的文本会在此区域展示，并用不同颜色标识，方便用户对比。

3.1.3 顶部操作栏
保存: 将当前编辑数据保存到后端。

导出为 PDF: 触发后端生成 PDF 文件的流程，并提供下载。

3.1.4 侧边栏调整参数
功能: 允许用户调整简历的全局样式，如主题颜色、字体风格、模块间距、字间距等。

交互: 调整参数后，预览区域实时更新，但这些参数不会作为用户编辑的数据保存，而是作为渲染时的样式属性。

3.2 后端功能模块
数据存储: 将前端发送的 JSON 数据持久化保存到数据库。

模板管理: 后端管理多个简历模板，每个模板对应一种排版和样式。

PDF 生成: 接收前端发送的 JSON 数据和用户选择的模板 ID，将数据注入到相应的模板中，并根据模块中的 style 字段进行样式渲染，最终生成并返回 PDF 文件流。\*\*后端将使用 Puppeteer 库来完成这一任务。具体流程如下：

动态生成 HTML：后端使用模板引擎（如 EJS）将 JSON 简历数据填充到预设的 HTML 模板中，生成完整的 HTML 字符串。

启动无头浏览器：使用 Puppeteer 启动一个无头（headless）的 Chromium 浏览器实例。

渲染页面：将生成的 HTML 内容加载到 Puppeteer 页面中，等待页面完全渲染。

生成 PDF：调用 page.pdf() 方法将渲染好的页面输出为 PDF 文件流。

返回文件：将 PDF 文件流作为 HTTP 响应返回给前端。\*\*

4. 关键交互逻辑
   4.1 编辑与预览同步
   用户在编辑区域修改内容。

前端捕获输入事件，并更新内部的 JSON 数据结构。

前端根据更新后的 JSON 数据，重新渲染预览区域。

4.2 模板与数据分离
前端: 仅负责接收用户编辑的数据（内容）。

后端: 负责管理模板（排版和样式）。

最终渲染: 当导出 PDF 时，后端将用户数据与选定的模板结合，生成最终文件。这种设计模式确保了数据的独立性，即使更换模板，用户数据也无需变动。

4.3 模块显示与隐藏控制
默认状态: 每个模块的默认显示/隐藏状态由模板 JSON 中的 visible 参数决定。

交互逻辑: 模块编辑区域会有一个独立的开关按钮（例如，显示/隐藏）来控制该模块在即时预览区域的可见性。

如果模板中 visible 为 false，则该按钮默认关闭，预览区域不显示该模块。

如果用户手动打开开关，即使模板中 visible 为 false，预览区域也应立即显示该模块。

持久化: 当用户点击“保存”时，前端会将用户当前的 visible 状态更新到 JSON 数据中，并将其保存到后端。

4.4 模块排序交互
数据更新: 当用户点击“上移”或“下移”按钮时，前端会重新排列 JSON 数据数组中的元素。

实时预览: 数组排序后，前端会立即重新渲染预览区域，以反映新的模块顺序。

持久化: 排序后的 JSON 数组将被保存到后端，以便下次加载时保留用户自定义的顺序。

4.5 模块内部条目排序交互
数据更新: 当用户在某个可重复模块（如教育背景）中点击条目的“上移”或“下移”按钮时，前端会重新排列该模块 items 数组中的元素。

实时预览: 数组排序后，前端会立即重新渲染预览区域，以反映新的条目顺序。

持久化: 排序后的 items 数组将被保存到后端，作为用户数据的一部分。

4.6 前端生成 HTML 并保存
前端生成 HTML：前端在渲染即时预览区域时，会根据 JSON 数据生成一个完整的 HTML 字符串。这个 HTML 字符串包含了所有模块的布局和样式，能够独立地在浏览器中正确显示。

保存到云端：当用户点击“保存”时，前端将不再仅仅发送 JSON 数据，而是将 JSON 数据和生成的 HTML 字符串一同发送给后端。

后端保存：后端在接收到请求后，将 JSON 数据（用于未来编辑）和 HTML 字符串（用于快速导出）分别存储到数据库中。

导出流程优化：当用户请求导出 PDF 时，后端不再需要进行复杂的模板引擎渲染，而是直接从数据库中读取预先存储的 HTML 字符串，并将其提供给 Puppeteer 进行 PDF 转换。

优点：该方案避免了前后端渲染差异，确保了所见即所得的导出效果，并且减轻了后端在导出时的计算负担。
