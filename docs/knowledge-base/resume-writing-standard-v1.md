# 标准简历知识库 v1

## 1. 事实与证据边界

简历优化必须遵守“证据边界四原则”：不虚构公司、岗位、学校或证书；不新增原文没有的数字；不改变时间与职责边界；不把团队成果冒充个人成果。缺少数据时，应使用待核实占位提示，而不是猜测结果。

## 2. 经历条目的推荐结构

每条经历优先采用“情境或目标—个人行动—方法或工具—可验证结果”的顺序。第一句话说明负责对象与目标，后续要点使用强动词开头，明确个人动作、协作边界和交付物。结果可以是效率、质量、收入、成本、规模、稳定性或用户体验，但必须来自候选人提供的事实。

## 3. 岗位匹配

岗位匹配不等于堆砌关键词。应先识别职位描述中的核心任务、必备能力和业务场景，再把候选人已有事实映射到相同语义。关键词必须出现在可验证的经历、项目或技能上下文中。与目标岗位弱相关且无法证明能力迁移的内容应压缩，而不是机械删除。

## 4. 技术岗位写法

技术经历应说明技术选择服务的业务目标、关键约束、个人负责范围和验证方式。仅罗列框架名称不足以证明能力。前端岗位可说明性能、可访问性、工程效率、组件复用和线上质量；后端岗位可说明可靠性、数据一致性、吞吐、延迟、安全与可观测性；算法岗位应说明数据、评估指标、基线和上线效果。

## 5. 产品与运营岗位写法

产品经历应交代用户问题、决策依据、方案权衡、推进角色和结果。运营经历应说明目标人群、渠道或策略、实验方法、转化链路和复盘。若结果属于团队，应明确个人贡献，例如负责分析、方案设计、跨团队推进或上线验收。

## 6. 校园与初级候选人

工作经验较少时，可以使用课程项目、竞赛、社团、志愿活动和个人作品证明能力。描述仍需体现目标、行动和产出，避免使用“认真负责、学习能力强”等无法验证的自我评价。课程名称本身不是成果，应补充所解决的问题和实际交付物。

## 7. 可读性与分页

中文正文不应通过无限缩小字号强塞一页。系统的一页适配可读性下限为 12px 字号、1.4 行高、12px 模块间距和 8px 条目间距。超过下限后仍无法容纳时，应保留自然分页。模块标题不得成为页尾孤行；同一条经历或项目应尽量整体换页，不在页面中间撕裂；长模块可以在条目之间跨页。

## 8. 输出检查

优化结果必须检查：是否引入未经证实的事实或数字；是否保留原始时间、主体和职责；是否与目标岗位有关；是否给出具体动作与结果；是否存在空泛形容词；是否出现标题孤行或条目跨页；PDF 是否成功生成且页数大于等于一。
