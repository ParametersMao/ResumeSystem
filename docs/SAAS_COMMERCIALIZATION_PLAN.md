# ResumeSystem SaaS 商业化升级计划

> 文档状态：待评审  
> 编写日期：2026-06-24  
> 目标：将当前“核心功能基本可用”的简历产品升级为具备多用户注册、套餐权益、计量扣减、支付闭环和运营能力的可商业化 SaaS。

## 1. 当前结论

当前项目已经完成了产品的核心能力层：

- 模块化简历编辑、实时预览、模板切换、版本管理。
- Puppeteer PDF 导出、照片上传、R2/本地对象存储。
- AI 润色、生成、诊断、LangGraph Agent、知识库与 RAG。
- C 端、中台、NestJS、MySQL、Agent、Qdrant 的 Docker 化部署。
- 基础账号密码登录、用户中心和中台数据统计。

当前完成度适合定义为：

> **“核心功能可稳定使用，具备内测和小范围试用条件。”**

### 当前必须优先修复的多用户安全问题

商业化之前不能直接沿用当前部分简历接口的访问方式：

- 简历创建、列表、详情、更新、删除和版本接口尚未全部使用 C 端 JWT Guard。
- 部分接口通过查询参数或请求体中的 `userId` 判断数据归属。
- 客户端提供的 `userId` 不可信，登录用户可以尝试修改该值访问其他用户数据。
- PDF 导出接口当前也缺少用户鉴权和额度关联。

必须统一改为：

- 所有用户资源接口强制 JWT 鉴权。
- `userId` 只从服务端解析后的 Token 获取。
- Service 查询始终包含 `resource.userId = currentUser.id`。
- 管理员访问用户数据使用独立的管理接口和审计日志。
- 增加跨用户访问自动化测试，确保返回 403/404。

这是从单用户内测进入多用户 SaaS 的 **P0 安全门槛**，优先级高于邮箱、微信和支付。

它还不能定义为商业 SaaS，主要缺口不是编辑器功能，而是以下商业基础设施：

- 邮箱验证注册、找回密码、第三方身份绑定尚未形成闭环。
- 微信扫码登录尚未接入。
- 套餐、订阅、订单、支付和退款不存在。
- 权益只存在 `ai_free_total / ai_free_used` 的早期雏形。
- PDF 导出、简历数量、版本数量、存储空间等没有统一额度控制。
- 缺少用户可理解的套餐页、用量页、账单页和升级入口。
- 中台不能配置套餐、赠送权益、处理订单和观察商业数据。
- 缺少商业上线所需的协议、注销、数据删除、支付对账与风控能力。

因此下一阶段的主线应从“继续堆功能”切换为：

> **身份体系 -> 权益与计量 -> 套餐 -> 支付 -> 运营 -> 合规与稳定性。**

## 2. 商业 SaaS 的完成标准

以后项目完成度建议使用以下五级标准，而不再只用“页面能打开、接口能调用”衡量。

### L1 功能原型

- 核心页面和主要交互存在。
- 可以使用模拟数据或单用户数据。
- 不要求完整权限、异常恢复和数据一致性。

### L2 基本可用

- 主要流程使用真实数据库和真实 API。
- 核心功能可稳定执行。
- 有基础登录、保存、导出和错误提示。
- 当前项目大致处于此级别。

### L3 产品可交付

- 多用户数据严格隔离。
- 关键接口有鉴权、限流、幂等和审计。
- 异常、空状态、超限、降级流程完整。
- 核心链路经过自动化测试和回归测试。
- 可部署、可备份、可恢复、可观测。

### L4 商业可运营

- 注册、登录、找回密码和账号绑定完整。
- 套餐、权益、计量、订单、支付、退款完整。
- 用户能查看套餐、剩余额度、账单和到期时间。
- 中台能配置套餐、调整权益、查询订单和处理异常。
- 有用户协议、隐私政策、退款规则和注销能力。

### L5 规模化 SaaS

- 支持灰度发布、渠道、优惠券、邀请裂变和企业套餐。
- 支持财务对账、发票、风控、成本和转化分析。
- 具备任务队列、缓存、高可用、容量规划和灾备。

本阶段目标应设为 **L4 商业可运营**，而不是直接追求 L5。

## 3. 产品与计费原则

### 3.1 编辑能力不按次数收费

简历编辑、自动保存、预览、模块排序和基础样式调整应保持不限次数。

原因：

- 编辑次数难以向用户解释。
- 自动保存会造成大量无感扣费。
- 限制编辑会破坏核心体验并降低付费转化。

应限制的是资源型或高成本能力：

- 可保存简历数量。
- 每份简历的历史版本数量。
- PDF 导出次数。
- AI Credit。
- 文件存储空间。
- 高级模板和高级 Agent 能力。

### 3.2 AI 使用统一 Credit，不直接暴露 Token

用户不需要理解模型 Token，应使用可理解的 AI 点数。

首版建议权重：

| AI 操作 | 消耗 Credit |
|---|---:|
| 单段润色 | 1 |
| 空内容生成 | 2 |
| 模块定向生成 | 2 |
| 全简历深度诊断 | 3 |
| 按岗位/JD 定向优化 | 4 |
| 全简历 Agent 重写 | 6 |

实际模型 Token、供应商费用和延迟仍写入内部成本账本，用于后续调价。

### 3.3 采用“权益账本”，不只在用户表增加计数字段

所有额度变化必须有来源、时间和关联业务：

- 套餐发放。
- 每月重置。
- 购买加量包。
- 运营赠送。
- 功能消费。
- 失败返还。
- 退款回收。
- 管理员调整。

任何扣减都应使用数据库事务和幂等键，禁止简单的“先查询剩余、再更新”，否则并发请求会超扣。

### 3.4 首版不做自动续费

第一阶段建议采用固定期限会员：

- 30 天求职会员。
- 90 天求职会员。
- 后续再增加年度会员。

固定期限支付的业务、退款和合规复杂度低于自动续费。待订单、对账和用户留存验证稳定后，再评估免密续费。

## 5. 用户身份体系

### 5.1 账号模型调整

`c_users` 应只代表平台用户主体，登录方式拆到身份表。

新增 `user_identities`：

| 字段 | 说明 |
|---|---|
| id | 身份记录 ID |
| user_id | 平台用户 ID |
| provider | password/email/wechat |
| provider_subject | 邮箱地址、微信 unionid/openid 等 |
| verified | 是否已验证 |
| provider_data | 非敏感扩展信息 JSON |
| create_time/update_time | 时间 |

唯一约束：

- `(provider, provider_subject)` 唯一。
- 一个用户可以同时绑定邮箱、密码和微信。

### 5.2 邮箱注册与登录

首版优先实现邮箱，它不依赖微信开放平台资质。

完整流程：

1. 输入邮箱。
2. 获取 6 位验证码。
3. 校验验证码并创建账号。
4. 设置密码或使用邮箱验证码直接登录。
5. 支持忘记密码。
6. 支持修改邮箱和绑定邮箱。

需要新增：

- `email_verification_codes` 表。
- 验证码哈希存储，不保存明文。
- 10 分钟有效、60 秒发送冷却。
- 单邮箱和单 IP 发送频率限制。
- 单验证码最多尝试 5 次。
- SMTP 连接测试、发送日志和失败重试。

### 5.3 微信扫码登录

建议在邮箱登录稳定后作为第二登录方式。

基本流程：

1. 前端请求创建微信登录会话。
2. 后端生成带随机 state 的二维码授权地址。
3. 用户扫码授权。
4. 后端用 code 换取微信身份标识。
5. 已绑定则直接登录；未绑定则创建用户或要求绑定已有账号。
6. 返回平台 JWT，不把微信 Token 直接交给前端。

必须处理：

- state 防 CSRF。
- 登录二维码超时。
- unionid/openid 的账号合并策略。
- 同一微信绑定多个账号的冲突。
- 用户主动解绑后至少保留一种可登录方式。

外部前置条件：

- 微信开放平台账号。
- 已审核的网站应用。
- 已备案的回调域名。

## 6. 权益与计量系统

### 6.1 核心数据表

建议新增：

#### `plans`

套餐定义，包括名称、价格、有效期、展示状态和排序。

#### `plan_entitlements`

套餐具体权益：

- `resume_limit`
- `version_limit_per_resume`
- `ai_credit`
- `pdf_export_limit`
- `storage_bytes`
- `premium_template_access`
- `agent_access`

#### `user_subscriptions`

用户当前或历史订阅：

- 套餐。
- 生效时间。
- 到期时间。
- 状态。
- 来源订单。
- 是否自动续费。

#### `user_entitlement_balances`

用户当前各项余额或上限。

#### `usage_ledger`

不可变的权益流水：

- user_id
- metric_code
- delta
- balance_after
- source_type
- source_id
- idempotency_key
- expire_at
- create_time

#### `usage_daily_summary`

按日聚合，供中台统计，避免每次扫描明细账本。

### 6.2 统一权益服务

新增 `EntitlementService`，业务模块不直接操作余额字段。

核心方法：

```ts
check(userId, metric, amount)
consume(userId, metric, amount, idempotencyKey, source)
grant(userId, metric, amount, expireAt, source)
refund(userId, idempotencyKey, source)
getSnapshot(userId)
```

### 6.3 功能接入点

| 功能 | 校验时机 | 扣减时机 |
|---|---|---|
| 新建简历 | 创建前检查存量上限 | 创建成功后无需流水扣减，按存量计算 |
| 保存版本 | 创建版本前 | 成功后按存量计算 |
| AI 调用 | 调用模型前预占 | 成功确认，失败释放/返还 |
| PDF 导出 | 生成任务前预占 | PDF 成功后确认，失败返还 |
| 文件上传 | 上传前检查空间 | 上传成功记录字节，删除后释放 |
| 高级模板 | 选择模板时 | 权限型权益，不按次扣减 |

### 6.4 重置策略

- 免费版 AI/PDF：自然月重置。
- 付费月卡：按订阅周期重置。
- 季卡：每 30 天生成一个权益周期，避免一次性发完后被滥用。
- 加量包：独立有效期，不随月度重置清零。
- 套餐到期：保留用户数据，只禁止新增超额资源。

套餐到期后若用户已有 10 份简历，而免费版只能保存 2 份：

- 允许查看、编辑和导出已有简历。
- 禁止新建，直到删除至上限或续费。
- 不得自动删除用户简历。

## 7. 支付与订单系统

### 7.1 首发支付方式

建议顺序：

1. 微信支付 Native 扫码。
2. 支付宝电脑网站支付。
3. 移动端 H5 支付。
4. 自动续费后置。

### 7.2 核心数据表

#### `orders`

- 平台订单号。
- user_id。
- product_type / product_id。
- 原价、优惠、实付金额。
- 订单状态。
- 支付渠道。
- 到期时间。
- create_time / paid_time / close_time。

#### `payment_transactions`

- 订单号。
- 第三方交易号。
- 请求金额。
- 支付状态。
- 回调原文摘要。
- 幂等键。

#### `refunds`

- 退款单号。
- 原订单。
- 退款金额。
- 退款原因。
- 状态。
- 第三方退款号。

### 7.3 订单状态机

```text
CREATED
  -> PAYING
  -> PAID
  -> FULFILLED

CREATED/PAYING
  -> CLOSED

PAID/FULFILLED
  -> REFUNDING
  -> REFUNDED
```

支付成功和发放权益必须分开记录：

- 支付回调只确认支付。
- 独立的履约事务发放订阅和额度。
- 即使服务重启，未履约订单也能补偿执行。

### 7.4 支付安全要求

- 服务端验签。
- 金额以服务端商品配置为准，不信任前端金额。
- 回调接口必须幂等。
- 同一订单不能重复发放权益。
- 定时主动查单，修复回调丢失。
- 每日自动对账。
- API Key、证书和商户密钥不得进入 Git。
- 支付日志不得记录完整敏感字段。

- 邮箱验证码登录。

- 微信扫码登录。
- 忘记密码。
- 用户协议和隐私政策勾选。

### 8.2 套餐与支付

- 套餐对比页。
- 当前套餐和到期时间。
- 支付二维码/支付跳转。
- 支付结果轮询。
- 订单列表。
- 退款规则入口。

### 8.3 用量中心

用户应能看到：

- AI Credit：已用/总量/重置时间。
- PDF：已用/总量/重置时间。
- 简历：当前数量/上限。
- 存储空间：已用/上限。

禁止只提示“操作失败”。

统一返回：

```json
{
  "code": "QUOTA_EXCEEDED",
  "metric": "AI_CREDIT",
  "required": 3,
  "remaining": 1,
  "resetAt": "2026-07-01T00:00:00+08:00",
  "upgradeUrl": "/pricing"
}
```

前端展示剩余额度、重置时间和升级入口。

## 10. 合规、安全与可靠性

### 10.1 商业上线前必须完成

- ICP 备案及阿里云新增接入。
- 用户协议。
- 隐私政策。
- 付费服务协议。
- 退款规则。
- 账号注销。
- 个人信息导出和删除申请。
- 邮件营销单独授权，不能默认勾选。

### 10.2 简历数据属于高敏感业务数据

简历可能包含：

- 姓名、电话、邮箱。
- 个人照片。
- 教育和工作经历。
- 求职意向。

需要：

- 数据库和对象存储访问最小权限。
- 日志脱敏。
- R2 文件使用不可预测 key。
- 私有文件优先使用短期签名 URL。
- 删除账号时异步清理简历、版本、照片和 AI 输入输出。
- 明确 AI 服务商会接收哪些内容，并提供必要提示。

### 10.3 稳定性

- 支付、扣费、导出采用幂等键。
- PDF 和 Agent 长任务逐步迁移到任务队列。
- MySQL 每日备份并定期恢复演练。
- Qdrant 和 R2 数据纳入备份策略。
- Sentry/日志告警覆盖登录、支付、AI、PDF 和存储。

## 11. 实施阶段与工期

以下为单人主开发的保守估算，可并行时缩短。

### 阶段 A：SaaS 数据基础，1.5 至 2 周

- 数据库迁移机制收口，停止运行时自动建表。
- 新增套餐、订阅、权益余额和权益流水表。
- 实现 `EntitlementService`。
- 将 AI 现有次数限制迁移到统一 Credit。
- 为新建简历、版本、PDF 和存储接入额度检查。

验收：

- 并发调用不会超扣。
- 功能失败会返还预占额度。
- 中台能看到用户完整权益快照和流水。

### 阶段 B：邮箱账号体系，1 至 1.5 周

- 用户身份表。
- 邮箱验证码注册和登录。
- 找回密码。
- SMTP 配置与测试。
- 发送频控和登录风控。
- 账号绑定与注销基础。

验收：

- 新用户可只使用邮箱完成注册、登录和找回密码。
- 邮箱不可重复占用。
- 验证码不可重放和暴力猜测。

### 阶段 C：套餐与用户侧产品化，1 至 1.5 周

- 套餐配置表和中台套餐管理。
- C 端套餐页、用量中心和升级入口。
- 统一超限错误码和弹窗。
- 到期降级策略。

验收：

- 中台修改套餐后，新订单使用新配置。
- 用户能清楚理解当前套餐、剩余额度和重置时间。

### 阶段 D：订单与支付，2 至 3 周

- 订单状态机。
- 微信 Native 支付。
- 支付回调验签和幂等履约。
- 支付结果轮询。
- 订单列表。
- 主动查单、关单和基础退款。

验收：

- 同一回调重复 10 次只发放一次权益。
- 回调丢失可通过主动查单补偿。
- 金额篡改无法支付低价订单。
- 支付成功后套餐和额度立即生效。

### 阶段 E：微信扫码登录，1 至 1.5 周

- 微信网站应用 OAuth。
- 二维码登录状态。
- 微信账号绑定、解绑和冲突处理。
- 与邮箱账号合并。

验收：

- 微信新用户可注册登录。
- 老用户可以绑定微信。
- 解绑后不会产生无法登录的孤儿账号。

### 阶段 F：商业上线收口，1.5 至 2 周

- 退款、对账和订单异常中台。
- 商业数据看板。
- 用户协议、隐私政策、注销和数据删除。
- 安全、性能、备份和恢复测试。
- 灰度发布和回滚方案。

总工期建议：**8.5 至 11.5 周**。

### 两个独立交付里程碑

#### 里程碑一：非付费 SaaS 基础

本里程碑先完成除付费外的全部用户服务：

- 多用户资源鉴权与数据隔离。
- 邮箱验证码注册、登录、找回密码。
- 用户名密码登录兼容。
- 用户档案、头像、账号安全、账号注销与数据归档。
- 微信扫码登录与已有邮箱账号绑定。
- 免费版权益、AI/PDF/简历/版本/存储额度。
- 用户用量中心和中台用户权益管理。
- 协议、隐私授权、数据导出与删除基础能力。

验收后，系统即使不收费，也应能够作为一个真实的多用户免费 SaaS 稳定运行。

#### 里程碑二：付费商业闭环

在里程碑一稳定后再建设：

- 付费套餐和商品。
- 订单、支付、退款和对账。
- 会员订阅和付费权益发放。
- 优惠券、加量包、发票及商业数据分析。

两个里程碑共享统一权益服务，但第一阶段不依赖支付系统即可独立上线。

## 13. 开工前需要确认的产品决策

以下内容需要产品负责人评审后冻结第一版：

1. 是否采用“免费版 + 30 天月卡 + 90 天季卡”。
2. 月卡 19.9 元、季卡 49.9 元是否适合目标用户。
3. 免费版是否允许每月 3 次无水印 PDF。
4. 免费版保存 2 份简历是否合适。
5. AI 是否采用统一 Credit 及建议权重。
6. 首发是否只做微信支付，还是同时做支付宝。
7. 微信登录是否可以晚于邮箱登录和支付上线。
8. 是否允许游客先编辑、注册后保存。
9. 用户注销后数据保留期和立即删除策略。
10. 是否暂缓自动续费。

## 14. 建议的第一版决策

为了尽快形成可收费闭环，建议：

- 登录：先邮箱，后微信。
- 支付：先微信 Native，支付宝紧随其后。
- 套餐：Free + 30 天月卡 + 90 天季卡。
- 计费：AI Credit + PDF 次数 + 简历存量 + 存储空间。
- 编辑：不限次数。
- 免费 PDF：每月 3 次，无水印。
- 自动续费：首版不做。
- 加量包、优惠券、发票、企业版：首版不做。
- 正式收费前完成 ICP、协议、隐私、退款和注销。

这条路线能够最大限度复用当前编辑器、AI、PDF、R2、中台和 Docker 基础，同时避免一开始就进入复杂的自动续费、企业计费和营销系统。
